## Overview

Generates an interactive HTML documentation site from the CSV data produced by `map_solutions_connectors_tables.py` and `collect_table_info.py`. The output consists of two main parts:

1. **Interactive index page** (`index.html`) вЂ” A single-page HTML application with DataTables.js providing filterable, sortable, searchable tables for all entity types (solutions, connectors, tables, content items, parsers, ASIM parsers).

2. **HTML entity pages** вЂ” Styled HTML versions of every markdown documentation page generated by `generate_connector_docs.py`, with consistent navigation and visual style.

The script can be invoked standalone or is automatically called by `generate_connector_docs.py` at the end of its documentation generation pipeline.

## Prerequisites

### Python Environment

- Python 3.7 or higher
- `markdown` package (required for HTML entity page generation)


The script reads the same CSV files as `generate_connector_docs.py`:

### Input Markdown Files

When generating HTML entity pages, the script expects markdown documentation files in the output directory, as produced by `generate_connector_docs.py`. These are converted to styled HTML pages.

## Running the Script

### Via generate_connector_docs.py (Recommended)

The interactive docs are generated automatically as part of the main documentation pipeline:

```bash
python generate_connector_docs.py \
    --output-dir "C:\path\to\docs" \
    --html-output-dir "C:\path\to\repo-root" \
    --html-docs-path "Solutions Docs/" \
    --html-index-url "https://user.github.io/repo/index.html"
```

### Standalone Usage

```bash
python generate_interactive_docs.py \
    --output-dir <docs-directory> \
    --input solutions_connectors_tables_mapping.csv \
    --connectors-csv connectors.csv \
    --solutions-csv solutions.csv \
    --content-items-csv content_items.csv \
    --tables-csv tables_reference.csv \
    --tables-overrides-csv tables.csv \
    --content-tables-csv content_tables_mapping.csv \
    --table-schemas-csv table_schemas.csv \
    --parsers-csv parsers.csv \
    --asim-parsers-csv asim_parsers.csv
```

To place `index.html` at a different location from the docs (e.g. for GitHub Pages):

```bash
python generate_interactive_docs.py \
    --output-dir <docs-directory> \
    --html-output-dir <repo-root> \
    --html-docs-path "Solutions Docs/" \
    --html-index-url "https://user.github.io/repo/index.html" \
    --input solutions_connectors_tables_mapping.csv \
    --connectors-csv connectors.csv \
    --solutions-csv solutions.csv \
    --content-items-csv content_items.csv \
    --tables-csv tables_reference.csv
```

### .nojekyll

A `.nojekyll` file is created alongside `index.html` to prevent GitHub Pages from running Jekyll. Jekyll's Liquid engine misinterprets `{{` sequences in Azure deployment template URIs found in generated docs, causing build failures.

### Features

- **DataTables.js**: All tables use [DataTables.js](https://datatables.net/) with Bootstrap 5 styling
- **Per-column filters**: Text input filters in a second header row for each filterable column
- **Click-to-filter**: Click any cell value to filter that column by the clicked text
- **Global search**: Quick filter input for full-text search across all columns
- **Clear All Filters**: A yellow notification bar appears when any filter is active, with a button to clear all filters at once

### Marketplace Fields in the Solutions Tab

The Solutions tab surfaces marketplace metadata sourced from the `mp_*` columns of `solutions.csv` (populated by `map_solutions_connectors_tables.py` from the Azure Marketplace API):

| Column | Source field(s) | Display |
|--------|-----------------|---------|
| Status | `mp_is_published`, `mp_is_preview`, `mp_is_stop_sell`, `is_deprecated` | Active / Deprecated / Unpublished badge |
| Publisher | `mp_publisher_display_name` (falls back to `publisher`) | Plain text |
| First Published | `mp_creation_date` | Formatted date |
| Popularity | `mp_popularity` | Bucketed icon: рџџў High / рџ”µ Medium / рџџЎ Low / вљЄ Very Low |

Other `mp_*` fields (summary, categories, keywords, ratings, pricing model, Microsoft product flag) are consumed by the static doc generator for the solution property table and are not displayed as separate columns in the interactive index.

- **Entity links**: All entity names link to their individual documentation pages (`.html` or `.md` depending on configuration)
- **Collection method links**: Method names in the Connectors tab link to their method documentation pages
- **Solution logos**: Displayed in the Solutions and Connectors tabs (loaded lazily)
- **Pagination**: Configurable rows per page (25, 50, 100, All) with 50 as default
- **Tab hash linking**: URL hash fragments (e.g. `index.html#connectors`) activate the corresponding tab. Supported on page load, hash change, and back/forward navigation (bfcache via `pageshow` event).
- **Loading overlay**: Displayed while data loads and DataTables initializes

## HTML Entity Pages

When `--html-output-dir` is set with a relative `--html-docs-path`, the script converts every `.md` file under the docs directory to a styled `.html` page. This enables GitHub Pages to serve the entire documentation site as HTML without relying on GitHub's blob view for markdown rendering.

### Conversion Process

1. All `.md` files under `--output-dir` are discovered via recursive glob
2. Each file is read and its title is extracted from the first H1 heading
3. Python `markdown` library converts the markdown to HTML using extensions: `tables`, `fenced_code`, `sane_lists`
4. Post-processing rewrites are applied:
   - Relative `.md` links в†’ `.html` links (for inter-page navigation)
   - Static index page links (e.g. `solutions-index.html`) в†’ `index.html#solutions` (redirect to interactive tab)
   - `README.html` home links в†’ `index.html`
   - The рџ”Ќ Interactive link from browse bars is removed (redundant in HTML context)
5. For table pages with a Schema section, the schema `<table>` is tagged with `id="schema-table"` and DataTables.js resources are injected

### Navigation

Each HTML entity page includes:
- A **navbar** linking back to the interactive index (via `--html-index-url` or relative path), with the link pointing to the relevant tab based on the page's subdirectory:
  - `solutions/` в†’ `#solutions`
  - `connectors/` в†’ `#connectors`
  - `tables/` в†’ `#tables`
  - `content/` в†’ `#content`
  - `parsers/` в†’ `#parsers`
  - `asim/` в†’ `#asim`
  - `methods/` в†’ `#connectors`
- A shared **page.css** stylesheet (written to `<html-output-dir>/css/page.css`) providing consistent styling matching the interactive index

### Schema Tables

Table entity pages that contain a "Schema" heading (`<h2>Schema...</h2>`) get special treatment:
- The schema `<table>` element receives `id="schema-table"` 
- DataTables.js CSS and JS are injected into the page
- A script initializes DataTables with per-column text filters on the schema table
- This allows filtering and sorting schema columns directly on the table page

## Integration with generate_connector_docs.py

The script is called at the end of `generate_connector_docs.py`'s main function via:

```python
from generate_interactive_docs import generate_interactive
generate_interactive(
    mapping_csv=...,
    connectors_csv=...,
    solutions_csv=...,
    content_items_csv=...,
    tables_csv=...,
    output_dir=...,
    html_output_dir=args.html_output_dir,
    html_docs_path=args.html_docs_path,
    html_index_url=args.html_index_url,
)
```

The `--html-output-dir`, `--html-docs-path`, and `--html-index-url` arguments are passed through from `generate_connector_docs.py`'s own CLI options.

When invoked through `generate_connector_docs.py`:
- The markdown docs have already been generated in `--output-dir`
- The interactive index reads the same CSV files as the doc generator
- HTML entity pages are generated from the markdown files just produced

## Synchronization with Static Indexes

The interactive HTML index and static markdown indexes (`solutions-index.md`, `connectors-index.md`, `tables-index.md`, `content-index.md`, `parsers-index.md`, `asim-index.md`) must present the same data with the same filtering rules.

When modifying either `generate_interactive_docs.py` or `generate_connector_docs.py`, ensure the following stay in sync:

- **Data filtering/inclusion rules**: Which entities to show, discovered vs. in-solution handling
- **Status classification logic**: Active/Deprecated/Unpublished determination
- **Icon usage and legend entries**: Same icons for same concepts
- **Link generation**: Entity links, collection method links, content source indicators
- **Special-case handling**: Placeholder name resolution (`<PlaybookName>`), "GitHub Only" solution name, Standalone non-linked solutions
- **Column additions or removals**: New data fields should appear in both views
- **Description cleanup**: Quote stripping, truncation rules

Similarly, when modifying **entity page content** in `generate_connector_docs.py`, ensure changes are reflected in both the markdown output and the HTML entity page rendering. In most cases, changes to markdown content flow through automatically (since HTML pages are derived from markdown). But changes involving link rewriting, heading structure, or HTML-specific formatting may need updates in `_generate_html_pages()`.

### Data Assembly Pipeline

The interactive index builds its own data rows from the CSV files, independent of the static doc generator. The data assembly functions mirror the static generator's logic:

| Function | Tab | Key Logic |
|----------|-----|-----------|
| `build_solutions_table_data()` | Solutions | Counts real connectors (excludes discovered), content items, marketplace popularity |
| `build_connectors_table_data()` | Connectors | Deduplicates by connector_id, uses `connectors.csv` for extended fields |
| `build_tables_table_data()` | Tables | Merges tables from connectors, content mapping, reference, and schemas |
| `build_content_table_data()` | Content | Resolves placeholder names from file paths, truncates descriptions |
| `build_parsers_table_data()` | Parsers | Filters TXT duplicates when YAML twin exists |
| `build_asim_table_data()` | ASIM | Excludes empty/placeholder parsers, categorizes as Unifying or Source |

### Performance

- HTML entity page generation processes ~10,000 markdown files in ~500-600 seconds
- A single `markdown.Markdown` converter instance is reused across all files (reset between files)
- Progress is reported every 200 files with elapsed time and estimated remaining time
- Files taking >5 seconds to convert are flagged with a warning
