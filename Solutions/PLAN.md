# Migration Plan: HTTP Data Collector API → Azure Monitor Logs Ingestion API

**Deadline:** September 14, 2026 (API retirement)  
**Affected solutions:** `Recorded Future`, `Recorded Future Identity`  
**Status:** Planning

---

## Background

The Azure Logic Apps managed connector `azureloganalyticsdatacollector` wraps the legacy
HTTP Data Collector API, which Microsoft is retiring on **September 14, 2026**. Every
data ingestion path in both of our solutions uses this connector. After the retirement
date, all affected playbooks will silently stop writing data to Log Analytics.

Reference: [Microsoft blog — Transitioning from the HTTP Data Collector API to the Log
Ingestion API](https://techcommunity.microsoft.com/blog/microsoftsentinelblog/transitioning-from-the-http-data-collector-api-to-the-log-ingestion-api%E2%80%A6what-doe/4403568)

---

## What Exactly Changes

### Old pattern (`azureloganalyticsdatacollector`)

```json
{
  "type": "ApiConnection",
  "inputs": {
    "body": "{ ... }",
    "headers": { "Log-Type": "RecordedFuturePortalAlerts" },
    "host": { "connection": { "name": "@parameters('$connections')['azureloganalyticsdatacollector']['connectionId']" } },
    "method": "post",
    "path": "/api/logs"
  }
}
```

### New pattern (Logs Ingestion API via HTTP action + Managed Identity)

```json
{
  "type": "Http",
  "inputs": {
    "method": "POST",
    "uri": "@{concat(parameters('DceEndpoint'), '/dataCollectionRules/', parameters('DcrImmutableId'), '/streams/', parameters('StreamName'), '?api-version=2023-01-01')}",
    "headers": { "Content-Type": "application/json" },
    "authentication": {
      "type": "ManagedServiceIdentity",
      "audience": "https://monitor.azure.com"
    },
    "body": [ /* array of objects */ ]
  }
}
```

### Key differences

| Concern | Old API | New API |
|---|---|---|
| Authentication | Workspace Shared Key (HMAC-SHA256) | Managed Identity / Entra ID OAuth |
| Table creation | Auto-creates table on first write | Table + schema must be pre-provisioned in ARM |
| Column naming | Auto-appends type suffixes (`_s`, `_d`, `_b`) | Columns stored exactly as declared in the DCR — **no suffixes** |
| Payload shape | Single object or raw array, no schema enforcement | Must be a JSON array; schema enforced by DCR |
| Infrastructure | Logic App connection resource only | DCE + DCR + role assignment required |
| Dynamic table name | `Log-Type` header set at runtime | DCR stream is fixed at deploy time |

---

## Breaking Change: Column Name Suffixes Removed

The legacy API auto-appended type suffixes to every column name:
- `email` → stored as `email_s`
- `login` → stored as `login_s`
- `domain` → stored as `domain_s`

The Logs Ingestion API stores columns **exactly as declared**. This means:

1. The DCR column declarations must use the bare names (`email`, `login`, `domain`).
2. **All KQL queries inside the playbooks that reference `_s`-suffixed columns must be
   updated.** Specifically:
   - `RFI-search-workforce-user`: queries `email_s`, `login_s`, `domain_s`
   - `RFI-search-external-user`: queries `login_s`, `domain_s`
3. Existing customers with historical data in old `_CL` tables will have both
   `_s`-suffixed columns (old data) and bare-named columns (new data). The migration
   guide must call this out.

---

## Approach Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Table name configurability | Keep as deploy-time parameter with default value | Preserves flexibility; DCR stream name defaults to match the table name parameter |
| Opaque payloads | Fully document all fields; use typed columns where possible, `dynamic` for deeply nested objects | Queryability + future-proofing |
| ARM template structure | All infrastructure in one `azuredeploy.json` per playbook (SOCRadar pattern) | Simpler deployment; no dependency chain between templates |
| `Package/mainTemplate.json` | Skip manual update — regenerated by `Tools/Create-Azure-Sentinel-Solution/createSolution.ps1` via CI | Only update individual `azuredeploy.json` files |
| Auth | Logic App System-Assigned Managed Identity | No secrets to manage; `Monitoring Metrics Publisher` role scoped to each DCR |
| Migration guide | Yes — document both fresh deployment and upgrade path | Existing customers must be informed |

## New ARM Resources Required Per Playbook

Every playbook needs these additions to its `azuredeploy.json`:

### 1. Logic App identity (add to existing `Microsoft.Logic/workflows` resource)
```json
"identity": { "type": "SystemAssigned" }
```

### 2. Data Collection Endpoint (one per playbook)
```json
{
  "type": "Microsoft.Insights/dataCollectionEndpoints",
  "apiVersion": "2024-03-11",
  "name": "[variables('DceName')]",
  "location": "[parameters('location')]",
  "properties": {
    "networkAcls": { "publicNetworkAccess": "Enabled" }
  }
}
```

### 3. Data Collection Rule (one per table)
```json
{
  "type": "Microsoft.Insights/dataCollectionRules",
  "apiVersion": "2024-03-11",
  "name": "[variables('DcrName')]",
  "dependsOn": [ "[variables('dceResourceId')]" ],
  "properties": {
    "dataCollectionEndpointId": "[variables('dceResourceId')]",
    "streamDeclarations": {
      "Custom-<TableName>_CL": {
        "columns": [ /* see per-table schemas below */ ]
      }
    },
    "destinations": {
      "logAnalytics": [{
        "workspaceResourceId": "[parameters('workspaceResourceId')]",
        "name": "workspace"
      }]
    },
    "dataFlows": [{
      "streams": ["Custom-<TableName>_CL"],
      "destinations": ["workspace"],
      "transformKql": "source",
      "outputStream": "Custom-<TableName>_CL"
    }]
  }
}
```

### 4. Log Analytics Table (one per table)
```json
{
  "type": "Microsoft.OperationalInsights/workspaces/tables",
  "apiVersion": "2022-10-01",
  "name": "[concat(parameters('workspaceName'), '/<TableName>_CL')]",
  "properties": {
    "schema": {
      "name": "<TableName>_CL",
      "columns": [ /* same columns as DCR streamDeclarations */ ]
    }
  }
}
```

### 5. Role Assignment — `Monitoring Metrics Publisher` on each DCR
```json
{
  "type": "Microsoft.Authorization/roleAssignments",
  "apiVersion": "2022-04-01",
  "scope": "[variables('dcrResourceId')]",
  "name": "[guid(variables('dcrResourceId'), variables('monitoringMetricsPublisherRoleId'))]",
  "dependsOn": [ "[variables('dcrResourceId')]", "[parameters('PlaybookName')]" ],
  "properties": {
    "roleDefinitionId": "[variables('monitoringMetricsPublisherRoleId')]",
    "principalId": "[reference(resourceId('Microsoft.Logic/workflows', parameters('PlaybookName')), '2019-05-01', 'Full').identity.principalId]",
    "principalType": "ServicePrincipal"
  }
}
```

### 6. New Logic App parameters (injected from ARM resource outputs)
```json

  "value": "[reference(variables('dceResourceId'), '2024-03-11').logsIngestion.endpoint]"

"DcrImmutableId": {
```

```
    latest_downloaded: datetime,

    compromise: { exfiltration_date: datetime },
    cookies: [{ dns, name, expiration, http: bool, secure: bool }],

      compromise: { antivirus[], computer_name, exfiltration_date, malware_file, os, os_username, timezone, uac },
      infrastructure: { ip: string },
```

## Migration Guide Outline (for both solutions)

Each solution's `MIGRATION_GUIDE.md` should cover:

### 1. What changed and why
Brief explanation of the API retirement and the new Logs Ingestion API.

### 2. Prerequisites for the new version
- No longer needed: workspace shared key
- New requirement: the deploying identity needs permission to create DCE, DCR, and role
  assignments in the target resource group
- New requirement: `Microsoft.Insights` resource provider registered in the subscription

### 3. What happens to existing data
- Historical data in `_CL` tables is unaffected; it remains queryable
- Old data has `_s`/`_d`/`_b`-suffixed column names; new data uses bare column names
- KQL queries that reference old suffixed columns will continue working against
  historical rows but will not match new rows — queries should be updated to use bare
  names or a union pattern during the transition period

### 4. Fresh deployment (new customer)
Standard deployment instructions pointing to the new template. No changes from the
customer perspective other than no longer needing a workspace key.

### 5. Upgrade path (existing customer)
1. Note any customized table name parameters before undeploying
2. Delete the existing playbook and its `azureloganalyticsdatacollector` API connection
   from the Azure Portal
3. Deploy the new version of the playbook template, providing `workspaceResourceId`
   and `workspaceName` (replacing `workspaceId` + `workspaceKey`)
4. Verify data flows into the table within one trigger interval
5. Update any saved KQL queries, alerts, or workbooks that reference `_s`-suffixed
   column names

### 6. Running old and new in parallel (optional)
Customers who want zero-gap coverage can deploy the new version alongside the old
before decommissioning the old. Both will write to the same `_CL` table name (old
writes with suffix columns, new writes with bare columns) until the old is removed.

### 7. Rollback
Old template remains functional until September 14, 2026. Customers can re-deploy the
previous version from source control if needed.
