## Introduction

This package contains the Microsoft Sentinel **Codeless Connector Framework (CCF)** artifacts for the **BitSight Risk Findings (via Codeless Connector Framework)** data connector.

The connector is designed to ingest BitSight risk-vector findings for a single portfolio company per connection into Microsoft Sentinel without using an Azure Function App. The package contains the connector UI definition, Content Hub solution metadata, custom table schema, Data Collection Rule (DCR), and the `RestApiPoller` configuration used by the connector.

The connector ingests **WARN** and **BAD** findings across all supported BitSight risk vectors returned by the BitSight Findings API. Filtering and normalization are handled by the DCR transform before the data is written to the custom table.

## Description

The connector brings BitSight finding data into Microsoft Sentinel for one BitSight company per connection. The connector does not restrict ingestion to a fixed risk-vector list. Instead, it requests findings from the BitSight company findings endpoint and allows all supported risk vectors returned by the API.

Only **WARN** and **BAD** findings are ingested. **FAIR**, **GOOD**, **NEUTRAL**, and **UNKNOWN** findings are filtered out by the DCR transform to reduce noise while retaining security-relevant findings.

## Prerequisites

1. A Microsoft Sentinel-enabled Log Analytics workspace.
2. A valid BitSight API token with access to the target company.
3. The BitSight **company GUID** to monitor.

No Data Collection Endpoint (DCE) needs to be created in advance — see the note below.

This template does **not** create a DCE resource. Instead, the DCR references an existing DCE through `dataCollectionEndpointId`. The current template assumes the DCE resource ID format below:

`/subscriptions/{subscription}/resourceGroups/{resourceGroupName}/providers/Microsoft.Insights/dataCollectionEndpoints/{workspace}`

If your DCE has a different name, update the `dataCollectionEndpointId` value in the template before deployment.

## Deployment

The connector deploys through the Microsoft Sentinel solution packaging flow generated by the Sentinel Create Solution v3 script.

1. Place the updated connector files in the `BitSightRiskFindingsLogs_ccf` folder.
2. Run the Sentinel Create Solution v3 packaging script.
3. Deploy the generated ARM template to the target subscription, resource group, and Microsoft Sentinel-enabled workspace.
4. Confirm that the data connector appears in **Microsoft Sentinel > Content management > Data connectors** with the display name **BitSight Risk Findings (via Codeless Connector Framework)**.

If an older BitSight CCF connection already exists in the workspace, disconnect or delete the old connection and reconnect after deployment to ensure the new `RestApiPoller` configuration is applied.

## Connecting a BitSight company

1. Go to **Microsoft Sentinel > Data connectors**.
2. Open **BitSight Risk Findings (via Codeless Connector Framework)**.
3. Provide:
   - **Company GUID** — the BitSight company to ingest findings for.
   - **BitSight API token** — entered as a secret.
4. Select **Connect**.

Each connection ingests findings for one BitSight company. To ingest multiple companies, create multiple connections, one per company GUID.

## How it works

- **Authentication**  
  The connector uses BitSight API token authentication and sends it through the poller authentication configuration as a Basic authorization header.

- **Endpoint**  
  The connector calls:

  ```http
  GET https://api.bitsighttech.com/ratings/v1/companies/{companyGuid}/findings
  ```

- **Risk-vector scope**  
  The connector does not send a `risk_vector` query parameter. This allows the API to return all supported BitSight risk vectors available for the configured company.

- **Time window**  
  Incremental polling uses:
  - `last_seen_gte`
  - `last_seen_lte`

  with a one-day (`1440` minute) polling window and `yyyy-MM-dd` formatting.

- **Response parsing and paging**  
  Findings are read from the BitSight response body and paging follows the `$.links.next` value.

- **Normalization and filtering**
  The DCR transform:
  - derives `normalized_risk_state`
  - keeps only `warn` and `bad`
  - allows all supported BitSight risk vectors returned by the API
  - maps the output to `BitsightRiskFindings_CL`
  - builds `finding_uid`
  - stores the normalized fields together with `details` and `raw` for investigation

## Validation

After connecting the data connector, allow at least one polling cycle to complete, then validate ingestion from Microsoft Sentinel Logs.

### Confirm latest data arrival

```kusto
BitsightRiskFindings_CL
| summarize Events=count(), DistinctFindings=dcount(finding_uid), Latest=max(TimeGenerated)
```

### Confirm only WARN and BAD findings are ingested

```kusto
BitsightRiskFindings_CL
| where TimeGenerated > ago(24h)
| summarize Events=count(), DistinctFindings=dcount(finding_uid) by normalized_risk_state
| order by Events desc
```

Expected states:

```text
warn
bad
```

### Confirm all returned risk vectors are collected

```kusto
BitsightRiskFindings_CL
| where TimeGenerated > ago(24h)
| summarize arg_max(TimeGenerated, *) by finding_uid
| summarize UniqueFindings=count(), Latest=max(TimeGenerated)
    by risk_vector, risk_vector_label, normalized_risk_state
| order by UniqueFindings desc
```

### Confirm polling cadence

```kusto
BitsightRiskFindings_CL
| take 10
```
